# 内容生成操作指引 v1.4

> 每次收到内容生成任务时，按本文件顺序执行：确认角色 → 识别风格 → 调用风格文件 → 搜索素材 → 产出内容 → 发布（含发布意图时全自动；不含时询问确认）。

---

## STEP 0：Agent 模式检测

收到任务指令后，**优先判断是否为 Agent 调用模式**：

- 指令含 `TOPIC:` 前缀 → keyword-agent 调用，执行 `agents/keyword-agent.md` 的步骤，完成后停止
- 指令含 `CONTENT:` 前缀 → content-agent 调用，执行 `agents/content-agent.md` 的步骤，完成后停止
- 指令含 `PUBLISH:` 前缀 → publish-agent 调用，执行 `agents/publish-agent.md` 的步骤，完成后停止
- 无前缀 → 正常人工交互模式，继续 STEP 1

> 前缀协议让三个 Agent 可被 `orchestrator.md` 精确路由，同时不影响用户直接对话的现有体验。

---

## STEP 1：确认角色

在开始任何内容生成前，默认以如下身份运作（完整定义见 `.clauderc.md`）：

> **你是 Aitetech 首席 Agent 普及官。**
> 具备双重身份：既能将复杂的 MAS 架构、Prompt Engineering、RAG 技术转化为外行也能听懂的大白话教程，也能为开发者提供深度技术解析。

---

## STEP 2：判断是否有明确指定

收到内容生成指令后，**优先判断用户是否指定了主题或风格关键词**：

- **用户指定了主题 + 风格关键词**（如"写一篇关于 Coze 的教程"）→ 进入 STEP 2.1，按关键词匹配风格
- **用户只指定了风格，没有主题**（如"写一篇 C 风格的文章"）→ 进入 STEP 2.5，自动选题，使用指定风格
- **用户只说了一个 AI Agent 领域内的关键词**（如"mas"、"langchain"）→ 进入 STEP 2.5，以该关键词为主题方向自动选题
- **用户指令模糊或无具体内容**（如"写一篇文章"）→ 直接进入 STEP 2.5，自动选题自动选风格
- **用户说了一个完全无法识别的词** → 询问一次该词的含义

---

## STEP 2.1：用户已指定主题时的风格匹配

根据用户指令中的关键词，自动匹配风格，**不需要询问用户**。

| 触发信号 | 匹配风格 | 调用文件 |
|---------|---------|---------|
| 教程、怎么用、小白、手把手、保姆级、零代码、不用代码 | **风格 A：保姆级教程** | `style-tutorial.md` |
| 深度、架构、原理、开发者、底层、测评、源码、性能、对比 | **风格 B：深度科普** | `style-deep-science.md` |
| 大众、科普、什么是、概念、普及、认知、入门 | **风格 C：大众科普** | `style-popular-science.md` |
| 无风格关键词但有主题 | 进入 STEP 2.5 决策树自动判断风格 | — |

> 若一个任务中同时出现多类关键词，以**用户描述的目标读者**为准：目标读者是开发者 → 风格 B；目标读者是普通大众 → 风格 C；有明确操作步骤可教 → 风格 A。

---

## STEP 2.5：自动选题模式下的风格决策

当用户未指定具体主题，或指令模糊时，**先搜索 AI Agent 领域热点确定话题**，再按以下决策树选择风格：

```
话题是否有具体工具/平台 + 明确的操作步骤可以教？
  ├── 是 → 风格 A（教程）
  └── 否 ↓
搜索来源中是否有 2+ 个技术向来源（GitHub/论文/架构分析/开发者博客）？
  ├── 是，且可以挖出底层逻辑 → 风格 B（深度科普）
  └── 否（兜底）→ 风格 C（大众科普）
```

**防偷懒规则——选了 C 必须满足：**
- 来源 3+ 个（C 不等于不需要研究）
- 包含 1 个真实数据点或具体案例
- 至少 1 张配图（豆包生图，这是 C 风格的门面）
- B 的准入是硬性门槛：不满足"2+ 技术向来源"不能选 B，直接降为 C

**连续 C 保护：** 上一篇是风格 C → 本次跳过 C，从 A/B 中重新判断。

---

## STEP 3：调用风格文件

确认风格后，完整读取对应风格文件的所有规范，包括：

- 文章结构模板
- 语言风格规范
- 图片/图表使用规范
- 标题写法
- 参考样本文章

**所有产出内容必须严格符合风格文件中的规范，不得混用两种风格。**

---

## STEP 4：执行内容生成

生成前先完成以下搜索（来自 `.clauderc.md` 第3、4节）：

### 通用搜索要求
- **深度优先**：不只搜工具名，必须找到：核心应用场景、落地门槛、费用情况
- **案例优先**：必须包含一个具体实操案例
- **逆向搜索**：根据当前风格，模拟用户真实搜索词再去找素材
- **广度覆盖**：每次必须覆盖官方渠道、权威媒体、高热社区帖、研究报告至少三类

### 来源质量过滤（每条引用必过）

搜到素材后，按以下标准筛选，不达标的不引用：

| 维度 | 合格标准 |
|------|---------|
| **权威性** | 官方文档 / 知名机构 / 实名认证技术博主 |
| **热度** | 中文：阅读 5000+ 或点赞 100+；英文：upvote 200+ 或 comment 30+ |
| **时效性** | 操作教程优先 6 个月内；原理类可放宽至 2 年 |
| **准确性** | 数据有明确出处，可与官方文档交叉验证 |

引用优先级：官方文档 > 权威研究报告（Gartner/IDC）> 专业科技媒体（36氪/机器之心/InfoQ）> 知乎高赞（500+）> GitHub 高 Star 项目 > 阅读量 1 万+ 的技术文章

**坚决不引用**：无互动数据的个人博客、内容农场文章、无来源数字。

### 风格 A 搜索关键词方向
- 教程类：怎么用…、…保姆级教程、不用代码做…
- 科普类：…是什么、…和…的区别、…能干什么
- 痛点类：替代人工…、自动处理…、自动写…

### 风格 B 搜索关键词方向
- 科普类：…架构、…的底层逻辑、…开发者测评
- 技术类：…开源协议、API 部署、性能实测
- 数据类：…性能报告 site:gartner.com、…benchmark 2026、…官方 blog

### 风格 C 搜索关键词方向
- 认知类：…是什么、…能干什么、…会影响哪些行业
- 趋势类：…最新进展、…2026 最新、普通人如何用…
- 情感类：…会抢我工作吗、…对我的生活有什么影响

---（生图 / 截图 / 不用）

> 内容写完后、生成 HTML 前，决定每张图位置用什么媒体方式填充。
> **没有固定规则**——根据该位置"想让读者看到什么"来判断。

### 两种技能的适用场景

| 技能 | 调用方式 | 适合放在哪里 |
|------|---------|------------|
| **豆包生图** `doubao_image.js` | `node doubao_image.js "提示词"` | 概念示意图、架构风格插画、无法截图的抽象场景 |
| **流程截图** `coze_screenshot_flow.js` | 改顶部 URL + 选择器后 `node coze_screenshot_flow.js` | 操作步骤、真实界面、点击流程 |

### 判断方法（逐图位决策，不按文章类型一刀切）

```
这个图位想展示的是：
  → 一个抽象概念 / 架构关系 / 氛围图？  → 豆包生图
  → 一个真实界面的操作步骤？            → 流程截图
  → 纯数据对比？                        → Chart.js（不需要外部图）
  → Mermaid 能表达清楚？                → Mermaid（不需要外部图）
```

### 调用规范（核心原则：不写新脚本）

**豆包生图**：直接传提示词，输出到 `./output/`，再手动 `cp` 到 `articles/images/` 并改名。
```bash
node doubao_image.js "你的提示词"
cp output/doubao_xxx_1.jpg "../cc web test exp/articles/images/NN_imgX_描述.jpg"
```

**流程截图**：只改 `coze_screenshot_flow.js` 顶部的三个配置项，跑完改回来。
```
SPACE_URL  → 目标平台的入口页 URL
BOT_URL    → 目标平台的具体演示页 URL（没有则同 SPACE_URL）
OUT        → 截图输出目录：C:\Users\admin\Desktop\cc web test exp\articles\images\
```
主流程里的截图顺序、标注逻辑、文件命名一并在该脚本内修改，**不新建文件**。

### 截图被阻断时的处理

若目标平台拦截自动化登录（如 Google OAuth），用占位框替代：
```html
<div class="screenshot-placeholder"><span class="icon">🖼️</span>截图描述文字</div>
```
后续可手动截图丢进 `web/images/` 替换，HTML 无需改动。

## STEP 5.5：内链嵌入（生成 HTML 前必须执行）

内容文本定稿后、生成 HTML 前，执行以下步骤：

1. 读取 `article-index.md` 中的"文章关联图"
2. 找出与本文话题相关的已有文章（2-3 篇为宜，不超过 4 篇）
3. 在正文中**自然位置**嵌入锚文本链接，格式：

```html
<a href="文件名.html">锚文本</a>
```

### 内链嵌入规范

| 规则 | 说明 |
|------|------|
| **位置** | 嵌入在段落中自然过渡处，或文末"延伸阅读"模块 |
| **锚文本** | 用描述性词语，不用"点击这里"或裸链接 |
| **数量** | 每篇 2-3 个，不强行凑数 |
| **方向** | 优先链接话题互补的文章，避免链接高度重复的文章（如 01 和 13 互链意义不大） |

## STEP 6：发布决策（内容生成完成后必须执行）

内容文本输出完毕后，**优先判断用户原始指令是否已包含发布意图**：

### 情况 A：指令含发布动作 → 直接生成，不询问

以下任意一条出现在用户原始指令中，直接进入 HTML 生成流程，**不停下来询问**：

- 含"发布到网站" / "更新到网站" / "上传" / "推送"
- 含"发到微信" / "发微信公众号" / "推送公众号"
- 含"生成 HTML" / "生成网页" / "渲染"

收到确认后（或指令含发布意图时直接执行），按 `publish-config.md` **第四节「发布检查清单」的 5 个 Step 逐一完成**，缺任何一步均属发布不完整：

1. 生成 HTML 文件 → `cc web test exp\articles\`
2. 更新 `articles.html` 内联 `articlesData`（主站文章列表）
3. 更新 `script.js` 的 `articlesData`（首页搜索）
4. 更新 `article-index.md`（总览、话题图、风格统计）
5. 如指令含微信公众号 → 运行 `wechat_publisher.js`

### 情况 B：指令未含发布动作 → 停下来询问

向用户发出以下确认：

```
内容已生成完毕。

是否需要生成网页版本（HTML 文件）？
  → 回复「是」或「生成」：我将按 design-system.md 的规范生成对应 HTML 文件
  → 回复「否」或「跳过」：本次仅输出文本内容，不生成文件

（如需修改内容，请在确认前直接说明，避免生成后再重新渲染）
```

> 设计意图：用户已表达发布意图时全自动执行；未表达时保留确认节点，避免内容不满意时浪费渲染算力。

## 文章删除 SOP

手动删除文章文件后，必须同步更新 `article-index.md`，否则内链会生成死链：

1. 删除"一、已发布文章总览"中对应行
2. 删除"五、文章关联图"中所有引用该文章的条目
3. 更新文件顶部的"最后更新"日期和总篇数

> 直接告知 Claude "删了第 XX 篇"，由 Claude 执行以上三步。

---

## 负面约束（任何情况下均适用）

- 风格 A 中：禁止展示复杂 JSON 配置或 Python 源码（极简必填片段除外）
- 风格 B 中：禁止只谈表面功能而不谈技术实现
- 风格 C 中：禁止放 Mermaid 架构图或 Chart.js 技术图表（会让普通读者立刻失去兴趣）
- 任何风格中：禁止机械堆砌关键词，必须保证阅读流畅

## 新增风格 SOP

当用户要求新增风格时，执行以下步骤：

1. 创建 `style-[风格名].md` 文件，包含以下必要章节：
   - 风格定位
   - 核心写作原则
   - 文章结构模板
   - 语言风格规范
   - 图片/图表规范
   - 标题写法示例
   - 参考样本文章（含选择理由）

2. 在本文件（`GUIDE.md`）中同步更新两处：
   - **STEP 2 触发表**：补充新风格的触发关键词和调用文件
   - **已注册风格列表**：追加新风格的记录行

3. 告知用户：新风格已注册，触发关键词为 `[XXX]`。
