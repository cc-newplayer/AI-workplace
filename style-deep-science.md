# 风格配置：深度科普风格

## 风格定位

面向有一定认知基础、想真正搞懂某个概念的读者。不追求"能立刻动手做"，而追求"读完之后真的理解了"。用类比、案例、对比拆解复杂概念，有观点、有逻辑、有深度，让读者既能转发又能复述给别人听。

---

## 核心写作原则

- **讲清"为什么"**：不只说"是什么"，要解释背后的原因和设计逻辑
- **类比先行**：每个抽象概念都要用一个生活化类比先铺垫，再讲技术细节
- **有观点**：作者要有明确立场，不做大而全的罗列，有取舍、有判断
- **对比揭差异**：多用"A vs B"的结构让读者理解事物的边界
- **结构驱动阅读**：小标题本身就是论点，读者扫一遍标题就能把握文章骨架

---

## 文章结构模板

```
# 标题（问题式或颠覆认知式，如"你以为的AI Agent和真正的AI Agent差在哪？"）

## 导语（200字以内）
- 用一个反直觉的现象或问题钩住读者
- 点明本文要回答的核心问题

## 一、背景/为什么这个问题值得讲
- 当前认知误区是什么
- 为什么现在是讲这个问题的好时机

## 二、核心概念拆解
- 先给出最精炼的定义（一句话）
- 再用类比展开
- 最后用技术语言精确描述

## 三、深入分析（文章核心，占全文60%）
- 子概念逐一展开
- 每个子章节：概念 → 类比 → 原理 → 局限性

## 四、横向对比
- 与相关概念/方案的本质区别
- 用表格或并列结构呈现

## 五、现实层面的观察
- 真实案例或数据支撑
- 当前的问题与挑战（不回避缺点）

## 六、结论/作者观点
- 明确表达立场
- 给读者一个"可以带走的判断框架"
```

---

## 语言风格规范

| 项目 | 要求 |
|------|------|
| 句子长度 | 长短结合，关键论点用短句单独成段以增强节奏感 |
| 人称 | 以"我们"为主，营造共同探索感 |
| 语气 | 理性克制，但不枯燥；有温度，不煽情 |
| 类比来源 | 优先用日常生活场景（餐厅点餐、团队分工、地图导航等）|
| 数据引用 | 必须注明来源，不用"据报道"等模糊表述 |
| 观点表达 | 用"我认为""值得注意的是"等显性标记区分事实与判断 |

---

## 写作节奏控制

```
[钩子段落]       → 反常识、强烈画面感、直接抛问题
    ↓
[建立认知锚点]   → 用类比建立基础理解，不超过3段
    ↓
[逐层深入]       → 每一层都在前一层基础上加一个维度
    ↓
[引入矛盾/张力]  → 讲清楚这个东西的局限或争议在哪
    ↓
[收口给判断]     → 给读者一个可以复述的结论性框架
```

## 标题写法示例

✅ 好标题：
- 「你以为 AI Agent 只是个聊天机器人？差得远呢」
- 「为什么说 ReAct 是目前最优雅的 Agent 推理框架」
- 「Multi-Agent 不是把多个 AI 堆在一起——一篇讲清楚多智能体的本质」

❌ 差标题：
- 「AI Agent 技术详解」
- 「关于多智能体系统的一些思考」

## 图表使用规范

深度科普文章的图不是截图，而是**概念图和对比图**：

| 图表类型 | 工具 | 适用场景 |
|---------|------|---------|
| **架构图 / 流程图** | Mermaid | 组件关系、ReAct 循环、决策分支 |
| **对比表** | HTML `<table>` | A vs B 差异、特性横评 |
| **横向条形图** | Chart.js | 多方案/框架性能对比 |
| **柱状图（前后对比）** | Chart.js | 效率提升、处理时效前后对比 |
| **环形图** | Chart.js | 市场份额、资源占比分布 |

**选型判断：**
- 数据有具体数字、适合量化比较 → 用 Chart.js（参考 `design-system.md` 5.8 节）
- 说明关系、过程、结构 → 用 Mermaid
- 无数据对比的文章不引入 Chart.js，不增加页面负担

图的作用是**辅助论点**，不是装饰。没有论点支撑的图不放。图表风格与文章保持一致：简约，不花里胡哨。

## 参考样本文章

**文章：《Multi-Agent System，一篇就够了》**

- 来源：知乎
- 链接：https://zhuanlan.zhihu.com/p/1928636720796136414

### 选择理由

这篇文章是本次整理的 20 篇中深度科普风格完成度最高的，具体体现在：

1. **概念边界划得清**：开篇就用"多个 Agent 堆在一起 ≠ 多智能体系统"这个反常识判断抓住读者，让人立刻想往下读
2. **类比质量高**：用"团队分工协作"类比 Orchestrator-Worker 模式，用"网状通信 vs 汇报制"类比不同架构，生活化且准确
3. **有观点有取舍**：作者明确指出网络型架构超过10个 Agent 就会拖慢性能，给出了有价值的工程判断，而不是罗列所有架构说"各有优劣"
4. **数据有来源**：引用 Anthropic 多 Agent 系统比单 Agent 性能高 90.2% 的实验数据，并注明出处
5. **标题即论点**：各小标题本身就是结论性判断，扫一眼就知道作者想说什么

### 可以学习的具体写法

- 导语段落用一个反常识陈述开头，不超过3句话就进入主题
- 每个架构模式先给一句话定义，再展开，最后写"适用场景"和"局限性"
- 案例部分用真实业务数字（如"订单交付从15天缩至8天"）而非泛泛描述
- 结论部分给出一个"选型判断框架"，让读者有东西可以带走
- 克制地使用粗体，只在段落中最关键的1-2个词上加粗，不滥用
