# 风格配置：保姆级教程风格

## 风格定位

面向零基础或初级读者，以"手把手带着做"为核心逻辑。读者读完后必须能独立完成文章中描述的操作，不留疑问、不跳步骤。语言口语化，像朋友在旁边指导，而非教科书讲解。

---

## 核心写作原则

- **步骤化**：所有操作必须拆解为编号步骤，每步只做一件事
- **截图优先**：每个关键界面、每次点击结果都要配截图，图下附说明
- **预期结果明确**：每步操作后告诉读者"此时你应该看到……"，用 `callout-info` 组件呈现
- **避免术语堆砌**：出现专业词汇必须当场解释，用括号或类比说明
- **主动预判障碍**：在可能出错的地方用 `callout-warning` 组件，小技巧用 `callout-tip` 组件；**禁止用 `<blockquote>` 替代 callout**

## 语言风格规范

| 项目 | 要求 |
|------|------|
| 句子长度 | 短句为主，一句话不超过30字 |
| 人称 | 全程用"你"，拉近距离 |
| 语气 | 积极鼓励，如"别担心，这一步很简单" |
| 专业词 | 首次出现必须加括号解释 |
| 数字 | 用具体数字代替模糊描述（"3步"而非"几步"）|

---

## 图片规范

- 每个主要步骤至少1张截图
- 截图中用**红框或箭头**标注操作位置
- 图片下方必须有一行说明文字
- 图片顺序与步骤严格对应

### 选择理由

这篇文章是本次整理的 20 篇中保姆级教程风格执行最到位的一篇，具体体现在：

1. **步骤拆解极细**：从 Docker 安装到模型配置，每个子步骤独立成段，无跳跃
2. **截图密度最高**：几乎每隔一步就有截图，并在图上标注了点击位置
3. **预判报错**：在 Docker 配置、国内镜像源等容易卡壳的地方提前写了注意事项
4. **结果导向清晰**：每个阶段结束后明确说明"此时你应该能看到 Dify 登录页面"之类的验证节点
5. **分上下篇**：上篇聚焦部署和工作流，下篇专门讲 RAG 知识库，结构清晰不贪多

### 可以学习的具体写法

- 用 callout-info / callout-warning / callout-tip 组件突出注意事项（**不用 `>` 引用块**）
- 命令行代码单独用代码块包裹，不混在正文里
- 每个大步骤前有一句"这一步我们要做的是……"的铺垫句
- 文章开头列出"本文涉及的技术栈"让读者心里有数
